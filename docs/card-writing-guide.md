# SillyTavern V4 卡片构建器完整写卡教程

> 适用对象：
> 
> - 想从零写一个角色卡的人
> - 想把已有卡片导入后继续改的人
> - 想做“角色卡 + 世界书 + 阶段解锁 + MVU 变量”的进阶用户
> - 想把小说、设定集、长篇资料整理成可玩的 SillyTavern 卡的人

---

## 1. 这是什么

这个项目不是单纯的“角色描述生成器”，而是一套偏完整的写卡工作台。

你可以把它理解成：

1. `角色区` 负责写角色本体
2. `世界书区` 负责写设定、势力、地点、规则、道具、事件
3. `小说世界书工坊` 负责把长文资料整理成带阶段门控的知识包
4. `MVU 变量卡` 负责做状态跟踪、JSONPatch 更新和状态面板
5. `试聊区` 负责验证“这张卡到底能不能跑起来”
6. `JSON 实时追踪` 和 `世界书内容监测` 负责查错

如果你以前只会写一段 `description`，这个工具的价值就在于：它把“写设定、注入规则、调参数、试运行、排错”放在了一页里。

---

## 2. 先理解四种核心产物

在开始之前，先把这四个概念分清：

### 2.1 角色卡主体

主要包括：

- `角色名`
- `角色描述`
- `第一条消息`
- `备选开场白`
- `作者注释`
- `头像`

这是角色的“人格与开场”。

### 2.2 世界书

世界书是“补充设定库”，常见内容有：

- 人物资料
- 地点
- 组织势力
- 规则体系
- 关键物品
- 剧情阶段信息

世界书不一定常驻显示给模型，而是按触发词、策略、深度、顺序等条件参与上下文。

### 2.3 小说世界书包

这是本项目的特色功能之一。

它不是简单把小说摘要塞进角色描述，而是把长篇小说资料整理成：

- `实体索引`
- `阶段化世界书条目`
- `揭露标记`
- `MVU 控制变量`

适合做“按原著进度推进”的卡。

### 2.4 MVU 变量系统

MVU 变量负责追踪运行中的状态，例如：

- 当前时间
- 当前地点
- 主角体力
- NPC 好感度
- 当前任务
- 线索、记忆、库存、阶段标记

它不是角色静态设定，而是对话过程中会变的状态层。

## 3. 界面分区说明

### 3.1 `AI 引擎`

作用：

- 配 API 地址和 Key
- 拉模型
- 开联网搜索增强
- 导入酒馆原生预设
- 一键生成角色和世界书骨架

你会经常用到这些输入：

- `API 接口地址`
- `API 密钥`
- `拉取模型`
- `联网搜索增强`
- `角色设定提示词`
- `世界书骨架提示词`
- `骨架模式条数`

### 3.2 `角色设定`

作用：

- 管理草稿
- 上传头像
- 写角色名、世界书名、角色描述、首条消息、备选开场白、作者注释

### 3.3 `小说世界书工坊`

作用：

- 导入 `.txt / .md / .json`
- 手动补摘录和上下文说明
- 选择门控模式
- 选择整理模式
- 选择抽取重点
- 生成并注入小说世界书

适合长文本整理，不适合“只写一个 OC”。

### 3.4 `世界书条目`

作用：

- AI 单条生成
- 新建条目
- AI 智能整理参数
- AI 批量补触发词
- 逐条编辑、重写、删除

### 3.5 `角色试聊`

作用：

- 用当前角色设定、世界书、预设直接聊天
- 看世界书有没有触发
- 看回复是否崩人设
- 调温度和回复长度
- 打开调试模式排查 Prompt

### 3.6 `MVU 变量节点`

作用：

- 生成变量系统
- 注入更新规则、输出格式、变量列表
- 预览变量
- 手动编排运行链路
- 追加单条变量

### 3.7 `JSON 实时追踪`

作用：

- 实时看卡片 JSON 结构
- 看字段注释
- 导入 `.json / .png`
- 导出 JSON
- 导出 PNG

### 3.8 `世界书内容监测`

作用：

- AI 深度审计世界书
- 看评分、问题和建议

## 4. 推荐的写卡流程

如果你是第一次用，建议按下面顺序走。

### 4.1 最稳的流程

1. 先配好 `AI 引擎`
2. 在 `角色设定` 里把主角基础信息填好
3. 在 `AI 引擎` 里先生成 `角色 + 世界书骨架`
4. 去 `世界书条目` 逐条展开、修改、补参数
5. 用 `角色试聊` 跑起来
6. 如果需要动态状态，再去 `MVU 变量节点`
7. 最后用 `JSON 实时追踪` 和 `世界书内容监测` 收尾检查

### 4.2 不推荐的流程

- 一开始就把所有设定都塞进 `角色描述`
- 世界书没整理就直接上 MVU
- 小说十几万字直接一次硬跑，不做分片和门控设计
- 完全不试聊，只看 JSON 觉得“应该没问题”

## 5. 从零写一张原创角色卡

这是最常见的场景。

### 5.1 第一步：配 AI 引擎

在 `AI 引擎` 面板中：

1. 填 `API 接口地址`
2. 填 `API 密钥`
3. 点击 `拉取模型`
4. 选择你要用的模型

如果你要做设定较强、世界书较多、后续还要跑小说整理，尽量选上下文和格式能力稳的模型。

### 5.2 第二步：写角色本体

在 `角色设定` 面板里填写：

- `角色名`
- `世界书名`
- `角色描述`
- `第一条消息`
- `作者注释`

建议这样分工：

- `角色描述` 写静态设定
- `第一条消息` 写开场氛围、说话方式、当前切入场景
- `作者注释` 写给卡片使用者的说明，不要把运行核心都堆这里

### 5.3 第三步：决定是手写还是 AI 起骨架

你有两种主路：

#### 路线 A：自己手写

适合：

- 你脑内设定已经非常完整
- 你不想让 AI 乱扩展
- 你在做严肃设定卡

做法：

1. 手写角色描述
2. 直接去 `世界书条目`
3. 一条一条创建世界书

#### 路线 B：AI 起骨架，再人工修

适合：

- 你有大方向，但设定还不完整
- 你想快速铺一层框架
- 你后续愿意自己收拾 AI 的产出

做法：

1. 在 `角色设定提示词` 写角色构想
2. 在 `世界书骨架提示词` 写世界观范围
3. 选择 `骨架模式` 条数
4. 点 `一键生成角色和世界书骨架`

这里生成的是骨架，不是最终成品。

## 6. 角色描述怎么写

### 6.1 角色描述适合写什么

建议重点写：

- 身份与定位
- 外貌与辨识特征
- 性格倾向
- 行为准则
- 背景经历
- 说话方式
- 价值观和禁区

### 6.2 角色描述不适合写什么

不建议把下面这些全部堆进描述：

- 全世界势力详解
- 每个地点的完整资料
- 全剧情时间线
- 大量可拆成世界书的条目
- 运行中的状态变量

这些更适合写进世界书或 MVU。

### 6.3 一个实用模板

```md
角色名：林雾

身份定位：
林雾是旧城区情报贩子兼二线修补师，表面上懒散圆滑，实际上对人与交易边界极度敏感。

外貌特征：
黑发，长期睡眠不足导致眼下发青，左耳有改造接口，常穿旧式防雨风衣，身上有淡淡机油味。

核心性格：
- 对陌生人先防备后试探
- 对弱者有底线保护欲
- 说话常留三分余地
- 很少直接承认自己的真实意图

行为逻辑：
- 优先规避无意义风险
- 只在收益、承诺或情感压力足够时主动出手
- 不喜欢失控局面，会本能地重新掌握谈话节奏

背景：
她曾在企业下层维修区工作，因牵连一次数据走私事件被迫离开体制，之后长期在灰色地带生存。

互动风格：
说话简短、带观察感，偶尔会用轻微嘲讽掩饰关心。
```

## 7. 第一条消息怎么写

第一条消息不是“角色简介再说一遍”，而是要完成三件事：

1. 立住声音
2. 立住当前场景
3. 给用户一个能接话的入口

### 7.1 好的第一条消息特点

- 有场景感
- 有角色语气
- 有当前动作
- 有留给用户回应的空间

### 7.2 常见错误

- 纯介绍，没有互动点
- 一上来信息太满
- 太像设定文，不像对话开场
- 直接讲太多剧透

### 7.3 一个示例

```md
雨水顺着旧铁棚边缘往下滴，林雾半蹲在摊位后面，指尖拨了拨一块还带余温的数据板。她抬眼看向你，目光先落在你的鞋跟，再落到你空着的手上。

“你不像来买零件的。”她慢吞吞地把板子扣回布下面，嗓音带着点刚熬完夜的哑意，“那就直说吧。你是来问路，问人，还是问一个最好别在这里提名字的东西？”
```

## 8. 世界书该怎么写

世界书是这个项目里最容易拉开质量差距的部分。

### 8.1 一条世界书通常应包含什么

一条好条目通常包含：

- `comment`：标题
- `content`：正文
- `keys`：触发词
- `strategy`：触发策略
- `position / depth / role / order / prob`

### 8.2 世界书适合写的内容

- 角色关系
- 地点特征
- 阵营立场
- 规则体系
- 职业、头衔、身份
- 道具用途
- 隐藏设定
- 阶段性认知边界

### 8.3 世界书正文写法

尽量写成“可直接注入模型的高密度设定”，而不是百科词条。

建议：

- 短而密
- 少空话
- 明确关系和限制
- 明确“会如何影响扮演”

不建议：

- 长篇剧情复述
- 含糊推测
- 大量废形容词
- 没有触发意义的流水账

### 8.4 `AI 单条生成` 的正确用法

当你已经有一些世界书条目时，可以在 `世界书条目` 面板用：

- `AI 单条生成（自动叠加）`

适合补：

- 某个地点
- 某个势力
- 某个专属武器
- 某个角色关系
- 某个规则说明

定向要求要具体，比如：

```md
写 1 条“黑雨市场”的世界书，强调它的交易规则、危险来源和林雾与这里的关系。
```

不要只写：

```md
写个地点
```

### 8.5 `融合角色描述` 和 `融合已有世界书条目`

这两个勾选项建议大多数时候都开着。

原因：

- 开 `融合角色描述`，新词条更贴角色本体
- 开 `融合已有世界书条目`，新词条更不容易和旧条目打架

如果你明确要写一个几乎独立的新模块，再考虑临时关掉。

## 9. 世界书参数怎么理解

这个项目支持较完整的世界书参数，不只是写正文。

### 9.1 `strategy`

常见有：

- `constant`
- `selective`

理解方式：

- `constant`：更像常驻规则
- `selective`：依赖触发词命中

经验上：

- 世界运行规则、底层法则，用 `constant`
- 人物、地点、组织、物品，多数用 `selective`

### 9.2 `keys`

触发词不要太偷懒。

人物条目建议包含：

- 本名
- 别名
- 称号
- 身份称呼
- 关系称呼

地点和势力条目建议包含：

- 全称
- 简称
- 别称
- 相关专名

### 9.3 `position / depth / role / order / prob`

如果你不想手调，可以先用：

- `AI 智能整理`

它会自动帮你整理：

- 插入位置
- 深度
- 顺序
- 概率

建议是：

1. 先把内容写对
2. 再做参数优化

不要本末倒置，一开始就只盯着参数。

## 10. 触发词生成怎么用

当你的世界书内容已经差不多了，但很多条目触发词还很粗糙时，用：

- `AI 触发词生成`

它适合解决：

- 触发词太少
- 只有一个名字，命中不稳定
- 条目本身没问题，但触发覆盖面不够

它不适合替代正文设计。

如果条目内容本身就是空的、虚的、错的，补多少触发词都没意义。

## 11. 小说转卡怎么用

这是本项目最有特色的一条路线。

如果你要做“按原著推进”的卡，不要把整本小说直接塞进角色描述，请优先走 `小说世界书工坊`。

### 11.1 适合什么场景

适合：

- 同人跑团式角色卡
- 按卷推进的原著卡
- 长篇剧情角色卡
- 大型设定集整理

不适合：

- 只做一个独立原创角色
- 只有几百字的短设定

### 11.2 导入方式

你可以：

- 拖入 `.txt`
- 拖入 `.md`
- 拖入 `.json`
- 或在“手动补充摘录”里贴片段

### 11.3 `门控模式` 怎么选

有三种：

- `阶段 + 揭露标记`
- `仅剧情阶段`
- `仅公开资料`

怎么理解：

- `阶段 + 揭露标记`：最完整，适合长篇、悬念和剧透控制强的作品
- `仅剧情阶段`：只管进度，不额外拆很多标记
- `仅公开资料`：只做无剧透设定包

### 11.4 `整理模式` 怎么选

现在新增了两种：

- `按剧情推进`
- `只做世界观`

区别非常重要：

#### `按剧情推进`

适合：

- 你要按原著推进
- 要保留前期、中期、后期认知差异
- 要保留真相揭露、关系变化、关键事件后果

输出倾向：

- 阶段化条目更多
- 会保留事件导致的状态变化
- 更适合“陪跑原著”

#### `只做世界观`

适合：

- 你只想拿原著世界观来跑原创剧情
- 你不想强绑定原著时间线
- 你只要人物、地点、规则、势力、物品

输出倾向：

- 更偏静态设定
- 更少时间线表达
- 更适合“借壳设定”

### 11.5 `抽取重点` 怎么选

可选重点包括：

- 人物
- 地点
- 势力
- 规则
- 物品

建议：

- 原著人物关系很重要的作品：人物 + 势力 + 地点
- 战斗或体系作品：规则 + 物品 + 势力
- 群像剧情作品：人物 + 势力 + 关键地点

### 11.6 `单次分片字数` 怎么调

这不是越大越好。

如果你是几十万字长文，尤其是 DeepSeek、某些公益站或上下文预算没那么宽的模型：

- 大分片容易在 `合并去重` 阶段炸 token
- 小分片调用更多，但更稳

建议思路：

1. 先用中等分片试跑
2. 如果合并阶段炸 token，就继续调低
3. 特别长的作品按卷、按篇、按阶段拆开做

### 11.7 小说工坊的推荐工作流

```md
第一轮：
先做“只做世界观”
目的：拿到稳定的人物 / 地点 / 势力 / 规则底盘

第二轮：
再做“按剧情推进”
目的：补阶段差异、真相揭露、剧情后果

第三轮：
在世界书管理里检查阶段按钮和揭露标记
确认哪些条目该在哪个阶段开放
```

### 11.8 小说工坊的常见坑

- 一次导入太大，模型或接口吃不下
- 想要“完全原著推进”，却选了 `只做世界观`
- 想做“自由二创”，却选了 `按剧情推进` 导致束缚太强
- 门控设计太复杂，自己都管不过来

## 12. MVU 变量卡怎么用

MVU 是进阶玩法，不是每张卡都必须有。

### 12.1 什么时候该上 MVU

当你想追踪这些东西时，MVU 很值：

- 时间流逝
- 位置变化
- 体力、金钱、库存
- NPC 情绪、关系、好感
- 任务、承诺、线索
- 阶段标记、一次性事件

如果你的卡只是短对话、轻陪聊、单场景互动，不一定非要上。

### 12.2 最简单的用法

在 `MVU 变量节点` 面板里：

1. 在 `变量需求` 写你想追踪的东西
2. 点 `生成变量卡并注入`
3. 看变量预览
4. 去试聊区实际跑

### 12.3 变量需求怎么写

建议写成“你想让模型长期维护什么”，比如：

```md
需要追踪：
- 世界时间和日期
- 主角当前地点
- 林雾当前情绪、行动、对主角的关系状态
- 主角持有的重要物品
- 当前任务和长期线索
```

### 12.4 MVU 生成后会出现什么

它会本地组装出这些模块：

- 变量结构脚本
- 变量初始化
- 更新规则
- 输出格式
- 变量列表
- 正则脚本

### 12.5 下方变量节点图是干嘛的

这是运行链路视图，用来帮助你理解：

1. Prompt 怎么组成
2. 世界书常驻条目怎么插入
3. `stat_data` 怎么喂给模型
4. 模型怎么输出 `<UpdateVariable>`
5. MVU 怎么解析 JSONPatch
6. 状态栏怎么刷新

如果你只是普通用户，可以不动这部分。

如果你是重度玩法用户，这里很有用。

### 12.6 什么时候要手动追加单条变量

当你整套变量差不多已经稳定，但临时少了一个字段时，用：

- `＋ 生成一条`

适合这种情况：

- 少一个商店库存
- 少一个秘密暴露值
- 少一个阶段标记
- 少一个特殊机制

不需要为了一两个字段把整套变量重做一遍。

---

## 13. JSON 实时追踪怎么用

这个面板非常重要。

### 13.1 它能做什么

- 看当前卡片 JSON
- 看字段注释
- 检查结构有没有问题
- 导入 JSON
- 导入 PNG 酒馆卡
- 导出 JSON
- 导出 PNG

### 13.2 `注释` 开关

建议一直开着。

它能帮助你判断：

- 某个字段写到哪一层才对
- 某个值是不是写错位置了
- 这条字段到底是注入 AI 的还是仅管理显示的

### 13.3 `编辑` 开关

适合小修小补。

如果你已经很熟悉字段结构，可以直接在这里改值。

如果你还不熟悉结构，建议优先通过面板改，而不是直接在 JSON 上乱改。

### 13.4 导入卡片的典型用法

当你有一张外部卡片时：

1. 在 `JSON 实时追踪` 点 `导入`
2. 选 `.json` 或 `.png`
3. 导入后到 `角色设定`、`世界书条目`、`MVU` 各区继续整理

这条路线很适合“拿旧卡重构”。

## 14. 角色试聊怎么做有效测试

很多卡在“看起来设定完整”和“实际能跑”之间差很远。

试聊不要只是发一句“你好”。

### 14.1 推荐测试顺序

1. 测开场是否自然
2. 测角色是否稳人设
3. 测世界书触发是否正常
4. 测长对话是否漂
5. 测关键场景时变量是否更新

### 14.2 建议测试的问题

可以这样测：

```md
1. 问一个角色绝不会答错的基础问题
2. 触发一个明显应该命中的地点或人物关键词
3. 故意推进一个状态变化，例如消费、受伤、移动、休息
4. 问一个处于“当前不该知道”的信息，看它会不会剧透
5. 连续聊几轮，看语气是否开始跑偏
```

### 14.3 `调试模式` 有什么用

开了之后更适合查这些问题：

- Prompt 里到底注入了什么
- 世界书有没有被触发
- 变量更新块有没有生成
- 回复长度和温度是否合适

---

## 15. 世界书内容监测怎么用

当你已经有一批世界书条目后，建议跑一次：

- `AI 深度审计`

它适合用来发现：

- 内容重复
- 条目空泛
- 触发意义弱
- 阶段设计混乱
- 参数不合理

建议时机：

1. 骨架铺完之后
2. 大改世界书之后
3. 小说工坊生成完成之后

## 16. 四条常见写卡路线

### 16.1 路线一：原创 OC 卡

适合：

- 纯原创角色
- 世界观不复杂
- 偏陪聊或中短剧情

做法：

1. 手写角色描述
2. AI 起少量世界书骨架
3. 单条补设定
4. 试聊

MVU 可选。

### 16.2 路线二：原创世界观剧情卡

适合：

- 角色和世界都重要
- 势力、地点、规则较多

做法：

1. AI 引擎生成骨架
2. 世界书分人物、地点、规则逐条重写
3. AI 智能整理参数
4. AI 触发词补全
5. 上 MVU 追踪时间、地点、关系、任务
6. 试聊

### 16.3 路线三：导入旧卡重构

适合：

- 你已有旧卡
- 但设定散、世界书乱、想升级

做法：

1. 在 `JSON 实时追踪` 导入 `.json/.png`
2. 去角色区修正文案
3. 去世界书区清理和重写
4. 必要时补 MVU
5. 审计和试聊

### 16.4 路线四：小说原著推进卡

适合：

- 长篇小说改卡
- 同人推进
- 阶段解锁和剧透控制要求高

做法：

1. 小说工坊先做 `只做世界观`
2. 再做 `按剧情推进`
3. 检查阶段与揭露标记
4. 同步到 MVU 变量节点
5. 试聊验证“未解锁信息是否泄露”

## 17. 常见问题与建议

### 17.1 为什么一键生成后很空

因为默认骨架模式就是“先铺框架，不一次写满”。

正确做法不是抱怨它短，而是：

1. 先看骨架方向对不对
2. 再逐条 AI 重写
3. 再补触发词和参数

### 17.2 为什么角色总漂

常见原因：

- 角色描述太空
- 世界书太杂乱
- 世界书参数没整理
- 试图让角色承担太多世界观解释任务
- 温度过高

### 17.3 为什么世界书不触发

先查：

1. `keys` 是否太少
2. 触发词是否写错
3. `strategy` 是否不合适
4. 参数是否被 AI 整理得太偏
5. 在试聊区世界书指示有没有变化

### 17.4 为什么小说工坊容易炸

常见原因：

- 文本太长
- 分片太大
- 合并阶段 token 超限
- 模型格式遵守差
- 公益站或代理接口稳定性差

解决思路：

1. 调低 `单次分片字数`
2. 按卷拆跑
3. 先做世界观，再做剧情推进
4. 换更稳的模型

### 17.5 为什么变量很多但不好用

因为变量“多”不等于变量“实用”。

好变量的标准是：

- 能被剧情自然更新
- 有明确用途
- 不会每轮瞎跳
- 不和其他变量重复

## 18. 一个完整实战建议

如果你不知道怎么开始，可以照这个模板做一次：

### 18.1 目标

做一张“赛博朋克情报贩子 + 城市灰区世界观 + 轻度状态追踪”的卡。

### 18.2 操作步骤

1. 在 `角色设定` 里写好角色名、描述、首条消息
2. 在 `AI 引擎` 里给两个提示词
3. 生成 `6` 条世界书骨架
4. 去 `世界书条目` 把人物、地点、组织、规则分别补满
5. 跑 `AI 智能整理`
6. 跑 `AI 触发词生成`
7. 在 `MVU` 里要求追踪：
   - 时间
   - 地点
   - 主角体力
   - 林雾情绪与关系
   - 当前任务
   - 重要线索
8. 生成变量卡并注入
9. 在 `角色试聊` 里测试五轮
10. 在 `世界书内容监测` 跑一次审计
11. 最后导出 PNG

这套流程基本就能做出一张“可跑、可修、可继续迭代”的卡。

## 19. 给不同人群的建议

### 19.1 如果你是新手

- 先别碰太复杂的 MVU
- 先学会把角色描述和世界书分开
- 先做一张 5 到 10 条世界书的小卡
- 多试聊，多改首条消息和世界书

### 19.2 如果你是设定党

- 少依赖一键成品
- 多用骨架 + 单条重写
- 多用审计和 JSON 实时追踪
- 参数整理放在内容写稳之后

### 19.3 如果你是长篇改卡用户

- 一定要用小说工坊
- 一定要做门控
- 一定要分阶段
- 一定要试“未解锁信息是否泄露”

### 19.4 如果你是机制党

- 上 MVU
- 让变量覆盖真实玩法需求
- 别做纯摆设字段
- 让每个变量都对应明确更新条件

## 20. 最后给一个总原则

这类卡的质量，通常不是由“某一段 description 写得多华丽”决定的，而是由下面四件事共同决定：

1. 角色本体是否清楚
2. 世界书是否分工明确
3. 状态层是否真实可维护
4. 试聊和排错是否认真做过

一句话总结这个项目的正确打开方式：

> 先把角色写清楚，再把世界拆开写，再把状态做成可维护结构，最后用试聊和审计把它打磨成能玩的卡。
