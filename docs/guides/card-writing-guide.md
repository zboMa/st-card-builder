# SillyTavern V4 卡片构建器完整写卡教程

> 适用对象：
> 
> - 想从零写一个角色卡的人
> - 想把已有卡片导入后继续改的人
> - 想做“角色卡 + 世界书 + 阶段解锁 + MVU 变量”的进阶用户
> - 想把小说、设定集、长篇资料整理成可玩的 SillyTavern 卡的人

---

## 1. 这是什么

这个项目不是单纯的“角色描述生成器”，而是一套偏完整的写卡工作台。

你可以把它理解成：

1. `角色区` 负责写角色本体
2. `世界书区` 负责写设定、势力、地点、规则、道具、事件
3. `小说` 七模块负责把长文资料拆章、生成角色设定/开场白、抽人物/世界书、蒸馏文风并同步到主卡
4. `MVU 变量卡` 负责做状态跟踪、JSONPatch 更新和状态面板
5. `试聊区` 负责验证“这张卡到底能不能跑起来”
6. `JSON 实时追踪` 和 `世界书内容监测` 负责查错
7. `右栏 AI 助手` 通过工具读写同一份卡片状态，协助检查、润色、跳转模块与试聊回流

如果你以前只会写一段 `description`，这个工具的价值就在于：它把“写设定、注入规则、调参数、试运行、排错”放在了一页里。

---

## 2. 先理解四种核心产物

在开始之前，先把这四个概念分清：

### 2.1 角色卡主体

主要包括：

- `角色名`
- `角色描述`
- `角色标签`（ST `tags` / `data.tags`）
- `第一条消息`
- `备选开场白`
- `作者注释`
- `头像`

这是角色的“人格与开场”。

### 2.2 世界书

世界书是“补充设定库”，常见内容有：

- 人物资料
- 地点
- 组织势力
- 规则体系
- 关键物品
- 剧情阶段信息

世界书不一定常驻显示给模型，而是按触发词、策略、深度、顺序等条件参与上下文。

### 2.3 小说工坊产出

这是本项目的特色功能之一。

它不是简单把小说摘要塞进角色描述，而是把长篇小说资料整理成：

- 章节结构
- 附录1 风格人物档案（可 AI 扩展）
- 非人物世界书草稿
- 可复用文风提示词

适合做“按原著资料整理再进主卡”的工作流。

### 2.4 MVU 变量系统

MVU 变量负责追踪运行中的状态，例如：

- 当前时间
- 当前地点
- 主角体力
- NPC 好感度
- 当前任务
- 线索、记忆、库存、阶段标记

它不是角色静态设定，而是对话过程中会变的状态层。

## 3. 界面分区说明

界面为三栏：`左侧边栏（顶部品牌区）` + `主内容区` + `右栏 AI 助手`。无全宽顶栏；侧栏顶部品牌区为「卡片构建器 / 一个简单的制卡器」，无版本或访客 tag。视口高度锁死，仅主区/助手对话/侧栏中间菜单内部滚动；侧栏三段式为顶固定品牌、中可滚业务菜单、底固定「配置」。侧栏分区如下：

- **角色卡制作**：角色卡管理 → 角色设定（右上角 **AI 引擎** 弹窗）→ 开场白 → 世界书条目 → 状态栏 → MVU 变量节点 → 正则 → 酒馆脚本
- **小说**：原始资料、拆章、角色设定、开场白、**小说分析**（含 G6 关系图谱）、**人物列表**、**世界书条目**、文风蒸馏；扫描/抽取为列表内降级路径；助手可 RAG 检索原文问答（见 `docs/architecture/novel-analysis.md`）
- **完成制作**：角色试聊 → JSON 实时追踪 → 世界书内容监测
- **配置**（固定底部）：AI 配置、提示词配置（含 AI 助手系统提示）

切换菜单不会卸载面板，未保存的输入会保留。可用地址 hash（如 `#character`、`#greetings`、`#chat`）深链到对应模块。窄屏下侧栏可折叠；助手栏在较窄视口会落到主区下方。

### 3.0a 右栏 AI 助手

- 默认与主工作区并排且占满右栏高度，不是侧栏菜单项；无折叠/收起。
- 支持多轮对话、工具轨迹；底部「快捷」浮层默认收起，点击展开，选预设后填入输入框并关闭（不直接发送）；点外或再点「快捷」也可关闭；发送/清空/撤销为小按钮横排。
- 标题左侧绿点表示就绪；会话保存在 `localStorage`（`st_v3_builder_assistant_session`），含消息与已注入 RAG 片段 ID（跨回合去重）。
- 小说 RAG 检索结果绑定到对应 user 回合（气泡仍只显示用户原文；模型历史含完整 RAG 块）；同一会话不重复注入已发送片段。
- 未配置接口/模型时发送仅 tip 提示，不发起对话。
- **小改自动应用**：例如微调角色名、单条世界书增改。
- **大改需确认**：删除条目、整段覆盖、批量补丁、整卡/骨架生成、小说草稿写入、MVU 注入等；确认区在对话下方折叠展示摘要。
- **撤销**：助手「撤销」对应上一补丁快照（`st_v3_builder_assistant_snapshots`）。
- 复用「AI 配置」中的接口与模型；可跳转侧栏模块；试聊后可用「试聊回流」分析并建议改卡。

### 3.0 `AI 配置`（配置区）

作用：横向三 Tab（默认进入 **API 配置**）：

- **API 配置**：API 地址 / Key、拉模型、联网搜索增强、Debug 开关
- **酒馆预设**：导入酒馆原生预设并勾选注入（勾选列表 `max-height: 540px`）
- **AI 引擎**：引擎相关选项；含 **标签生成上下文字数**（默认 12000 / 12k，供角色标签 AI 截断世界书等）

仅拆布局；存盘字段语义不变（可新增 `tagContextChars`）。一键生成在「角色设定」右上角 **AI 引擎** 弹窗。

### 3.0b `提示词配置`（配置区）

作用：

- 编辑各生成环节的系统/核心提示词模板
- **Tab 分类**：角色卡制作 / 世界书 / 状态栏·MVU / 小说 / AI 助手；切换 Tab 保留未保存编辑；可「保存全部」「恢复本页默认」「全部恢复默认」
- **描述体系**（`promptCanon.mjs`）：公共块统一驱动——内容维（身份/外貌/性格/关系/动机/钩子等）+ NSFW 维（人物/世界）+ 分步推断 + 输出契约 + 反空话；AdultMode 开关约束私密字段
- 默认值与代码内置一致；覆盖项写入 `localStorage`（`st_v3_builder_prompts`）
- 不改变动态拼接逻辑（角色信息、预设、搜索结果等仍按原流程注入）
- AI 助手分组：`assistantSystem`（含 `{{toolList}}`）、`assistantReactHint`、`assistantChatFeedback`

### 3.1 `AI 引擎`（角色设定弹窗）

入口：**角色设定** 面板右上角「AI 引擎」按钮，打开居中弹窗（Esc / 遮罩 / 关闭可退出）。

作用：

- 填写阶段 1 / 2 / 3 用户提示词（角色设定、世界书骨架、开场白）
- 设置骨架条数
- 一键生成角色、世界书骨架与开场白（**阶段 3 始终执行**）
- 阶段 2 / 3 会将前序生成结果注入提示词作参考，减少重复并便于衔接

你会经常用到这些输入：

- `[阶段1] 角色设定提示词`
- `[阶段2] 世界书骨架提示词`
- `[阶段3] 开场白提示词`（留空则按角色与骨架自动构思）
- `骨架模式条数`

API / 搜索 / 预设 / 标签上下文字数请到「配置 → AI 配置」对应 Tab。系统提示词含 `charGen`（阶段1）、`wbSkeleton`（阶段2）、`greetingGen`（阶段3），可在「提示词配置」调整。

### 3.1b `角色卡管理`

作用：

- 以预览卡片网格展示本地全部角色卡（封面/头像、卡名、更新时间、当前激活标记）
- **默认一行 4 列**；窄屏依次降为 3 / 2 / 1 列
- 右上角：新建 + 导入并排（无全局导出）
- 每张卡底部操作并列：复制、重命名、导出 JSON、导出 PNG、删除（导出**该卡**，ST 格式；**不含**小说工坊）
- 点击某卡（封面或卡名区域）：切换为当前卡并跳转「角色设定」

存储仍使用浏览器本地多卡草稿（`st_v3_builder_drafts` / `st_v3_builder_current_id`），不改变卡片字段语义。深链：`#card-manager`。

### 3.2 `角色设定`

作用：

- 编辑**当前**角色卡：上传头像、角色名、世界书名、**角色标签**、角色描述、作者注释
- 头像高清图（长边 ≤2048px）与封面缩略图（≤512px）存 **IndexedDB**；草稿 JSON 仅存 `avatarInIdb` 标记并在管理页显示封面；旧内嵌 `avatarBase64` 加载时自动迁入 IDB
- 角色标签以紧凑 chip +「**AI 生成** / 输入框 / 添加」横排增删（回车亦可）；导入优先读 `data.tags`，否则读顶层 `tags`；导出同时写入两处；助手 `get/set_character_fields` 的 `tags` 与之同步
- **AI 生成**：用角色设定 + 开场白 + 部分世界书（按「AI 配置 → AI 引擎」字数上限截断，默认 12k）生成短中文标签，**合并**进现有 tags（不清空用户已有）；任务中心可停；未配 AI 时 tip 提示
- 仅编辑表单（无「当前：xxx / 返回管理」顶栏）；多卡切换请到「角色卡管理」
- 右上角 **AI 引擎** 打开三阶段一键生成弹窗（见 §3.1）

多卡列表/新建/删除已迁至「角色卡管理」；主开场白与备选开场在「开场白」模块，字段语义与导出不变。

### 3.2b `开场白`

作用：

- 编辑主开场白（SillyTavern `first_mes` / `data.first_mes`）
- 管理备选开场列表（`data.alternate_greetings`）
- 支持增删、排序备选开场

与角色设定、AI 生成、导入导出共用同一套字段，不改变导出能力。

### 3.3 `小说`（主路径 + 降级）

左栏平铺：原始资料 → 拆章 → 角色设定 → 开场白 → **小说分析** → **人物列表** → **世界书条目** → 文风蒸馏。

视觉与主卡「角色设定」「开场白」等一致（共用 `.panel` / `.form-group` / `.btn`）。小说侧 view 为 `novel-character-setup` / `novel-greetings`，与主卡 `character` / `greetings` 分离。世界书条目与主卡「世界书条目」共用折叠预览 + 居中弹窗编辑交互。

作用：

- 导入 `.txt / .md / .json`，补充 Context；全局仅保留模式/并行/召回预算（分片字数在拆章与各抽取模块）
- 拆章后供后续模块共用（本模块配置单片字数）
  - **布局**：上方拆分配置 + 批量工具栏固定；仅章节列表内部滚动
  - **顶栏批量**：合并所选、启用/禁用所选、导出选中、删除所选
  - **行内**：左勾选 + 标题/字数，右横排图标（预览/对半/重命名/调序/启停/导出/删除）
  - **预览**：点标题或预览图标 → 居中遮罩弹窗看正文（取消/遮罩/Esc 关闭）；禁止列表内嵌大段正文
  - **禁用样式**：禁用章浅红背景与边框
- **角色设定生成**（`novel-character-setup`）：可从实体库选人；生成时优先 RAG 召回原文，无命中回退字数/章节范围预览；经任务中心写入 `charName` / `wbName` / `charDesc` / `creatorNotes`
- **开场白生成**（`novel-greetings`）：同上（实体+RAG / 范围回退）+ 开场白数量；写入 `first_mes` + `alternate_greetings`（主开场 1 + 备选 N−1）；未配 AI 时 tip 提示
- 人物**逐步**分片扫描（每步注入已扫描人物作参考，可补 aliases/identity）+ **AI 扩展**；每条可同步到角色设定 / 世界书；档案居中弹窗编辑
  - **布局**：上方配置/操作固定；仅人物列表内部滚动（对齐拆章）
  - **分片**：可选「按字数」（默认 8k）或「按章节」（每 N 章一次，默认 N=1）
  - **预估**：扫描按钮显示 `约 N 次 · 扫描全书`（与真实扫描同一分片逻辑；改方式/数值即时重算；**逐步**执行非并行；**每步刷新列表可预览**；提示词要求尽量产出 **aliases**）
  - **扩展**：点击后先弹窗展示原文摘录与字数，确认再执行；**优先原文**，未写明字段据已有信息**合理虚构补全**（禁止留「原文未提及」）；NSFW 含身体/敏感点/反差/XP/内心等；召回用 name+aliases；结果回写实体库
  - **同步落卡**：多人仅首人写 `charName`，其余 merge 追加区块；YAML 跳过空/「原文未提及」；`syncStatus` 与实体双向一致，未写入不标已同步
  - **停止**：扫描全书走任务中心；侧栏任务列表可停；进行中按钮「扫描中…」+ disabled；取消后提示「已取消」
- 非人物世界书**逐步**分片抽取（每步完成后将已抽取条目注入下一步参考，避免重复并可补充完善）；折叠行预览；点编辑打开居中弹窗；行内 **AI 扩展**；同步到主世界书
  - **策略标签**：列表「常驻/可选」为彩色小球 tag（蓝=常驻，绿=可选）；导出枚举仍为 `constant` / `selective`
  - **布局**：分片方式/数值与冲突策略同行；操作行顺序为 AI 抽取 / 新建 / 清空 / 同步所选→主世界书 / 包含成人设定；搜索紧贴列表；世界书名在角色设定
  - **分片 / 预估**：同人物模块（本模块独立配置）；AI 抽取按钮显示 `约 N 次 · AI 抽取`（**逐步**执行 N 步，非并行批量；**每步刷新列表可预览**；要求产出 **keys** 触发词）
  - **扩展**：标题+触发词匹配原文，按 AI 扩展召回预算扩写描述；同样先弹窗确认
  - **同步**：实体条目 + 未入库草稿按 comment 去重后一并写入主世界书
  - **停止**：AI 抽取走任务中心可停；逐步循环可中断；取消后提示「已取消」
- **小说分析**（`novel-analyze`）：RAG 索引（章节变更后提示过期；完整分析仅过期时重建）+ 骨架/丰满/关系；**NSFW/NTL** 仅在「原始资料·全局处理配置」开关（分析/世界书/文风不再各挂勾选）；失败片可「重跑失败」；「NSFW→状态栏草案」只生成不写入；页内 **G6** 图谱；清空图谱仅清关系，保留实体
  - 成人开：**分步推断**——骨架即出人物 NSFW 草稿 + 物品/地点/设定 `attrs.adult` + `nsfw` 实体；丰满/关系/抽取用新信息修订（原文→互证→虚构）；质量门校验人物 NSFW 与条目成人维；文风互喂含人物/nsfw/条目成人摘要
- 文风蒸馏后同步为标题固定「文风」的主世界书条目；勾选「包含情欲文风」时蒸馏结果须含结构化 **NSFW 文风指令**（节奏/身体粒度/敏感点/内心/反差/禁用套路等）；蒸馏任务可在侧栏任务中心停止

本地状态按角色卡 `cardId` 分桶（`novelWorkshopV3:card:{cardId}`）：**IndexedDB 优先**，localStorage 同名键与旧全局/V2 在首次加载时迁入 IDB；切卡加载对应桶，新建卡为空；导出角色卡 JSON/PNG **不含**小说内容；导入卡不会错误改绑小说桶。助手小说工具读写当前卡桶。

适合长文本整理，不适合“只写一个 OC”。

### 3.4 `世界书条目`

作用：

- AI 单条生成
- 新建条目
- AI 智能整理参数
- AI 批量补触发词
- 逐条编辑、重写、删除

布局：标题行右上小号横排「单条生成 / 智能整理 / 补全触发词 / 新建」；其下仅搜索筛选与条目列表。AI 配置只在居中遮罩弹窗内（确认执行原逻辑；取消/遮罩/Esc 关闭），**禁止**把配置表单内嵌进列表上方。搜索命中为**单列紧凑行**（标题 + 常驻/可选 tag + 命中字段 + 一行高亮摘录），显示命中数；无命中时短提示；**整行可点**跳转并展开对应条目。主列表：**点击条目行展开详情预览**（内容/触发词等），再点或「收起」折叠；**点「编辑」或「新建」打开居中弹窗**改字段（保存/取消关闭），列表不再行内大表单。策略在 UI 显示为中文 tag（常驻/可选/向量化，彩色小球），导出字段仍为英文枚举。**条目列表占满剩余高度并内部滚动**。整理完成后的预览/状态条可在列表上方短暂展示，不属于配置表单。

### 3.5 `角色试聊`（完成制作）

作用：

- 用当前角色设定、世界书、预设直接聊天
- 看世界书有没有触发
- 看回复是否崩人设
- 调温度和回复长度
- 打开调试模式排查 Prompt

布局：上方配置固定；中间对话区（世界书触发标签 / 调试 Prompt / 消息）统一内滚，标签过长不撑破整页；下方输入固定。标题行右上为指示器 → 调试 →「重置」「重新生成」。

### 3.6 `状态栏`

作用：

- 左：紧凑分步配置；右：大预览（随**排版方案** + **开启模块**联动重绘）

## 4. 推荐的写卡流程

如果你是第一次用，建议按下面顺序走。

### 4.1 最稳的流程

1. 先在「配置 → AI 配置」配好接口与模型
2. 在 `角色设定` 里把主角基础信息填好
3. 在 `AI 引擎` 里先生成 `角色 + 世界书骨架`
4. 在 `开场白` 里润色主开场与备选开场
5. 去 `世界书条目` 逐条展开预览、弹窗修改、补参数
6. 需要动态状态时，去 `状态栏` 分步生成变量设计（覆盖 MVU）并注入展示脚本
7. 需要微调运行链路或单条变量时，再去 `MVU 变量节点`
8. 需要微调正则或酒馆助手脚本时，到 `正则` / `酒馆脚本`
9. 在「完成制作」用 `角色试聊` 验证，再用 `JSON 实时追踪` 和 `世界书内容监测` 收尾

### 4.2 不推荐的流程

- 一开始就把所有设定都塞进 `角色描述`
- 世界书没整理就直接上 MVU / 状态栏
- 小说十几万字直接一次硬跑，不做分片和门控设计
- 完全不试聊，只看 JSON 觉得“应该没问题”

## 5. 从零写一张原创角色卡

这是最常见的场景。

### 5.1 第一步：配 AI 配置

在「配置 → AI 配置」的 **API 配置** Tab 中：

1. 填 `API 接口地址`
2. 填 `API 密钥`
3. 点击 `拉取模型`
4. 选择你要用的模型

如果你要做设定较强、世界书较多、后续还要跑小说整理，尽量选上下文和格式能力稳的模型。

### 5.2 第二步：写角色本体

在 `角色设定` 面板里填写：

- `角色名`
- `世界书名`
- `角色描述`
- `作者注释`

再到 `开场白` 模块填写：

- `主开场白`（`first_mes`）
- `备选开场白`（`alternate_greetings`，可选）

建议这样分工：

- `角色描述` 写静态设定
- `主开场白` 写开场氛围、说话方式、当前切入场景
- `作者注释` 写给卡片使用者的说明，不要把运行核心都堆这里

### 5.3 第三步：决定是手写还是 AI 起骨架

你有两种主路：

#### 路线 A：自己手写

适合：

- 你脑内设定已经非常完整
- 你不想让 AI 乱扩展
- 你在做严肃设定卡

做法：

1. 手写角色描述
2. 直接去 `世界书条目`
3. 一条一条创建世界书

#### 路线 B：AI 起骨架，再人工修

适合：

- 你有大方向，但设定还不完整
- 你想快速铺一层框架
- 你后续愿意自己收拾 AI 的产出

做法：

1. 在 `角色设定提示词` 写角色构想
2. 在 `世界书骨架提示词` 写世界观范围
3. 选择 `骨架模式` 条数
4. 点 `一键生成角色和世界书骨架`

这里生成的是骨架，不是最终成品。

## 6. 角色描述怎么写

### 6.1 角色描述适合写什么

建议重点写：

- 身份与定位
- 外貌与辨识特征
- 性格倾向
- 行为准则
- 背景经历
- 说话方式
- 价值观和禁区

### 6.2 角色描述不适合写什么

不建议把下面这些全部堆进描述：

- 全世界势力详解
- 每个地点的完整资料
- 全剧情时间线
- 大量可拆成世界书的条目
- 运行中的状态变量

这些更适合写进世界书或 MVU。

### 6.3 一个实用模板

```md
角色名：林雾

身份定位：

外貌特征：
黑发，长期睡眠不足导致眼下发青，左耳有改造接口，常穿旧式防雨风衣，身上有淡淡机油味。

核心性格：
- 对陌生人先防备后试探
- 对弱者有底线保护欲
- 说话常留三分余地
- 很少直接承认自己的真实意图

行为逻辑：
- 优先规避无意义风险
- 只在收益、承诺或情感压力足够时主动出手
- 不喜欢失控局面，会本能地重新掌握谈话节奏

背景：
她曾在企业下层维修区工作，因牵连一次数据走私事件被迫离开体制，之后长期在灰色地带生存。

互动风格：
说话简短、带观察感，偶尔会用轻微嘲讽掩饰关心。
```

## 7. 第一条消息怎么写

第一条消息不是“角色简介再说一遍”，而是要完成三件事：

1. 立住声音
2. 立住当前场景
3. 给用户一个能接话的入口

### 7.1 好的第一条消息特点

- 有场景感
- 有角色语气
- 有当前动作
- 有留给用户回应的空间

### 7.2 常见错误

- 纯介绍，没有互动点
- 一上来信息太满
- 太像设定文，不像对话开场
- 直接讲太多剧透

### 7.3 一个示例

```md
雨水顺着旧铁棚边缘往下滴，林雾半蹲在摊位后面，指尖拨了拨一块还带余温的数据板。她抬眼看向你，目光先落在你的鞋跟，再落到你空着的手上。

“你不像来买零件的。”她慢吞吞地把板子扣回布下面，嗓音带着点刚熬完夜的哑意，“那就直说吧。你是来问路，问人，还是问一个最好别在这里提名字的东西？”
```

## 8. 世界书该怎么写

世界书是这个项目里最容易拉开质量差距的部分。

### 8.1 一条世界书通常应包含什么

一条好条目通常包含：

- `comment`：标题
- `content`：正文
- `keys`：触发词
- `strategy`：触发策略
- `position / depth / role / order / prob`

### 8.2 世界书适合写的内容

- 角色关系
- 地点特征
- 阵营立场
- 规则体系
- 职业、头衔、身份
- 道具用途
- 隐藏设定
- 阶段性认知边界

### 8.3 世界书正文写法

尽量写成“可直接注入模型的高密度设定”，而不是百科词条。

建议：

- 短而密
- 少空话
- 明确关系和限制
- 明确“会如何影响扮演”

不建议：

- 长篇剧情复述
- 含糊推测
- 大量废形容词
- 没有触发意义的流水账

### 8.4 `AI 单条生成` 的正确用法

当你已经有一些世界书条目时，可以在 `世界书条目` 面板用：

- `AI 单条生成（自动叠加）`

适合补：

- 某个地点
- 某个势力
- 某个专属武器
- 某个角色关系
- 某个规则说明

定向要求要具体，比如：

```md
写 1 条“黑雨市场”的世界书，强调它的交易规则、危险来源和林雾与这里的关系。
```

不要只写：

```md
写个地点
```

### 8.5 `融合角色描述` 和 `融合已有世界书条目`

这两个勾选项建议大多数时候都开着。

原因：

- 开 `融合角色描述`，新词条更贴角色本体
- 开 `融合已有世界书条目`，新词条更不容易和旧条目打架

如果你明确要写一个几乎独立的新模块，再考虑临时关掉。

## 9. 世界书参数怎么理解

这个项目支持较完整的世界书参数，不只是写正文。

### 9.1 `strategy`

导出/存储仍用英文枚举：

- `constant`（界面显示「常驻」，蓝点 tag）
- `selective`（界面显示「可选」，绿点 tag）
- `vectorized`（界面显示「向量化」）

理解方式：

- `constant`：更像常驻规则
- `selective`：依赖触发词命中

经验上：

- 世界运行规则、底层法则，用 `constant`
- 人物、地点、组织、物品，多数用 `selective`

### 9.2 `keys`

触发词不要太偷懒。

人物条目建议包含：

- 本名
- 别名
- 称号
- 身份称呼
- 关系称呼

地点和势力条目建议包含：

- 全称
- 简称
- 别称
- 相关专名

### 9.3 `position / depth / role / order / prob`

如果你不想手调，可以先用：

- `AI 智能整理`

它会自动帮你整理：

- 插入位置
- 深度
- 顺序
- 概率

建议是：

1. 先把内容写对
2. 再做参数优化

不要本末倒置，一开始就只盯着参数。

## 10. 触发词生成怎么用

当你的世界书内容已经差不多了，但很多条目触发词还很粗糙时，用：

- `AI 触发词生成`

它适合解决：

- 触发词太少
- 只有一个名字，命中不稳定
- 条目本身没问题，但触发覆盖面不够

它不适合替代正文设计。

如果条目内容本身就是空的、虚的、错的，补多少触发词都没意义。

## 11. 小说转卡怎么用

这是本项目最有特色的一条路线。

如果你要做“按原著推进”的卡，不要把整本小说直接塞进角色描述，请优先走左栏 **小说** 七模块。

### 11.1 适合什么场景

适合：同人跑团、按卷推进原著卡、长篇剧情卡、大型设定集整理。  
不适合：只做一个独立 OC、只有几百字的短设定。

### 11.2 推荐工作流

```md
1. 原始资料：导入文本 + Context；设好并行 / AI 扩展预算（默认 30000）；「重置并清空结果」只清产出草稿，保留原文与配置
2. 拆章：配置本模块单片字数 → 自动拆章；禁用无关章；点标题/预览 → 弹窗看正文
3. 角色设定：填角色名 → 选字数或前 N 章 → 看预览 →「生成角色设定」写入当前卡（任务中心可停）
4. 开场白：填角色名与数量 → 看预览 →「生成开场白」写入主开场 + 备选（任务中心可停）
5. 人物列表：优先查看分析实体 → 丰满所选 / 档案编辑 → 同步；或降级「扫描全书」+ 行内 AI 扩展；长任务可停
6. 世界书条目：优先查看分析实体 → 类型筛选 / 丰满所选 → 同步；或降级「AI 抽取」+ 行内扩展；抽取可停
7. 小说分析页可查看关系图谱（G6）；清空图谱不影响人物/世界书实体
8. 文风蒸馏：配置抽样分片 → 蒸馏（可选 NSFW 文风段）→ 同步为世界书「文风」条目；蒸馏可停
```

### 11.3 处理模式

- `剧情延续`：偏剧情与关系变化
- `世界观重建`：偏静态设定底盘

### 11.4 AI 扩展注意点（人物 + 世界书条目）

- 人物：角色名 + 别名匹配原文；世界书条目：标题 + 触发词匹配；都不塞全书
- 预算不够时按首现/共现/分散章抽样（默认 30000，见原始资料）
- UI 必先弹窗确认摘录与字数；助手工具可直跑跳过弹窗
- 人物档案对齐附录1；世界书扩写写入该条 `content`
- 扩展后在本模块同步，避免误覆盖主卡

### 11.5 分片方式怎么调

- **拆章 / 文风**：仍按字数档位（4k～64k）。
- **人物 / 世界书 / 分析**：各自可选
  - **按字数**（默认，跨章 packing）：按启用章顺序贪心拼入当前片，加入下一章将超过分片字数则开新片；单章超长则章内切开。空章不计。`N` = 实际片数
  - **按章节**：`N = ceil(有文本启用章数 / 分片章节数)`
- 次数会即时显示在「约 N 次」按钮上；分片字数越大，跨章合并越多、次数越少。长文、上下文偏紧时先中等分片试跑；特别长的作品可按卷拆开或改用按章节。

### 11.6 常见坑

- 未拆章就去生成角色设定/开场白或扫人物/世界书（会被门控禁用）
- 生成角色设定/开场白会直接覆盖当前卡对应字段，生成前先确认范围与角色名
- AI 扩展前别名不全，召回命中过少
- 一次勾选太多章导致扫描成本过高
- 同步时选了「覆盖」却没检查主卡已有描述

## 12. MVU / 状态栏变量怎么用

MVU 是进阶玩法，不是每张卡都必须有。整套变量设计入口在 **状态栏**；`MVU 变量节点` 负责链路编排与变量查看/编辑（含单条追加、清除）。

### 12.1 什么时候该上 MVU

当你想追踪这些东西时，MVU 很值：

- 时间流逝
- 位置变化
- 体力、金钱、库存
- NPC 情绪、关系、好感
- 任务、承诺、线索
- 阶段标记、一次性事件

如果你的卡只是短对话、轻陪聊、单场景互动，不一定非要上。

### 12.2 最简单的用法（状态栏）

在 `状态栏` 面板里：

1. 选人数（分段：单人用当前卡角色；多人 AI 识别世界书人物并可勾选；默认只识别女角色；入选后人人同套变量）
2. 选预设短卡，按需开关模块；需要时打开 NSFW（关则身体模块整组隐藏）
3. 工具行点「生成并覆盖 MVU」（变量先于排版）
4. 选排版短卡（一对一视觉主题，**一主题一文件、结构互异**；15 美学族 × 单人/多人 = 30 套，按人数过滤；或选「自定义」AI 排版）
5. 确认预览后底栏「注入状态栏」
5. 各步用固定底栏「下一步」推进；「上一步」为次要描边5. 去试聊区实际跑

模块勾选与「刷新预览」会即时按开启模块重绘右侧预览。

### 12.3 预设与模块怎么选

建议按题材选预设，再微调开关：

- 常规：亲密度/好感、信任、关系阶段、情绪、行动、地点、着装、物品、金钱、任务、记忆摘要、事件芯片、属性条、时间天气（多人已去掉配角摘要；入选角色人人同套详字段）
- NSFW 开：小穴/双乳/美腿/美脚/屁穴/内心想法，以及口腔、敏感带、高潮/快感、体液、露出、调教标记、性经验摘要、当前性行为状态
- 「重置」可一键恢复预设默认模块

### 12.4 生成后会出现什么

状态栏生成会覆盖写入 MVU，并本地组装：

- 变量结构脚本
- 变量初始化
- 更新规则
- 输出格式
- 变量列表
- 正则脚本（也可在「正则」模块继续编辑）
- 酒馆助手脚本（也可在「酒馆脚本」模块继续编辑）
- 状态栏前端展示脚本 `[状态栏]前端展示`

### 12.5 下方变量节点图是干嘛的

这是运行链路视图，用来帮助你理解：

1. Prompt 怎么组成
2. 世界书常驻条目怎么插入
3. `stat_data` 怎么喂给模型
4. 模型怎么输出 `<UpdateVariable>`
5. MVU 怎么解析 JSONPatch
6. 状态栏怎么刷新

如果你只是普通用户，可以不动这部分。

如果你是重度玩法用户，这里很有用。

### 12.6 什么时候要手动追加单条变量

当你整套变量差不多已经稳定，但临时少了一个字段时，在 `MVU 变量节点` 用：

- `＋ 生成一条`

适合这种情况：

- 少一个商店库存
- 少一个秘密暴露值
- 少一个阶段标记
- 少一个特殊机制

不需要为了一两个字段把整套变量重做一遍。

### 12.7 从卡推定变量（规则合并）

当你已经配好状态栏模块或开启了恶堕/成人层，但不想整套 AI 重生成时：

1. 打开 `MVU 变量节点`
2. 点「从卡推定变量」
3. 在预览列表勾选候选（已有路径默认不勾）
4. 「应用合并」只写设计；「应用并注入」会合并后走现有注入链路

推定来源包括：状态栏 `moduleFlags`/`paths`、恶堕进度、同意边界、亲密度/体力/着装等基线字段。助手只读工具：`infer_mvu_variables`。

## 13. JSON 实时追踪怎么用

这个面板非常重要。

### 13.1 它能做什么

- 看当前卡片 JSON
- 看字段注释
- 检查结构有没有问题
- （导入 JSON/PNG 在「角色卡管理」右上角；导出 JSON/PNG 在各预览卡底部）

### 13.2 `注释` 开关

建议一直开着。

它能帮助你判断：

- 某个字段写到哪一层才对
- 某个值是不是写错位置了
- 这条字段到底是注入 AI 的还是仅管理显示的

### 13.3 `编辑` 开关

适合小修小补。

如果你已经很熟悉字段结构，可以直接在这里改值。

如果你还不熟悉结构，建议优先通过面板改，而不是直接在 JSON 上乱改。

### 13.4 导入卡片的典型用法

当你有一张外部卡片时：

1. 在 `角色卡管理` 右上角点 `导入`
2. 选 `.json` 或 `.png`（写入当前卡；不改小说分桶绑定）
3. 导入后到 `角色设定`、`世界书条目`、`MVU` 各区继续整理
4. 需要导出时在对应预览卡底部点 `JSON` / `PNG`

这条路线很适合“拿旧卡重构”。

## 14. 角色试聊怎么做有效测试

很多卡在“看起来设定完整”和“实际能跑”之间差很远。

试聊不要只是发一句“你好”。

### 14.1 推荐测试顺序

1. 测开场是否自然
2. 测角色是否稳人设
3. 测世界书触发是否正常
4. 测长对话是否漂
5. 测关键场景时变量是否更新

### 14.2 建议测试的问题

可以这样测：

```md
1. 问一个角色绝不会答错的基础问题
2. 触发一个明显应该命中的地点或人物关键词
3. 故意推进一个状态变化，例如消费、受伤、移动、休息
4. 问一个处于“当前不该知道”的信息，看它会不会剧透
5. 连续聊几轮，看语气是否开始跑偏
```

### 14.3 `调试模式` 有什么用

开了之后更适合查这些问题：

- Prompt 里到底注入了什么
- 世界书有没有被触发
- 变量更新块有没有生成
- 回复长度和温度是否合适

## 15. 世界书内容监测怎么用

当你已经有一批世界书条目后，建议跑一次：

- `AI 深度审计`

它适合用来发现：

- 内容重复
- 条目空泛
- 触发意义弱
- 阶段设计混乱
- 参数不合理

建议时机：

1. 骨架铺完之后
2. 大改世界书之后
3. 小说工坊生成完成之后

## 16. 四条常见写卡路线

### 16.1 路线一：原创 OC 卡

适合：

- 纯原创角色
- 世界观不复杂
- 偏陪聊或中短剧情

做法：

1. 手写角色描述
2. AI 起少量世界书骨架
3. 单条补设定
4. 试聊

MVU 可选。

### 16.2 路线二：原创世界观剧情卡

适合：

- 角色和世界都重要
- 势力、地点、规则较多

做法：

1. AI 引擎生成骨架
2. 世界书分人物、地点、规则逐条重写
3. AI 智能整理参数
4. AI 触发词补全
5. 上 MVU 追踪时间、地点、关系、任务
6. 试聊

### 16.3 路线三：导入旧卡重构

适合：

- 你已有旧卡
- 但设定散、世界书乱、想升级

做法：

1. 在 `角色卡管理` 导入 `.json/.png`
2. 去角色区修正文案
3. 去世界书区清理和重写
4. 必要时补 MVU
5. 审计和试聊

### 16.4 路线四：小说原著推进卡

适合：

- 长篇小说改卡
- 同人推进
- 阶段解锁和剧透控制要求高

做法：

1. 小说工坊先走「小说分析」+ 世界书条目整理
2. 再做 `按剧情推进`
3. 检查阶段与揭露标记
4. 同步到 MVU 变量节点
5. 试聊验证“未解锁信息是否泄露”

## 17. 常见问题与建议

### 17.1 为什么一键生成后很空

因为默认骨架模式就是“先铺框架，不一次写满”。

正确做法不是抱怨它短，而是：

1. 先看骨架方向对不对
2. 再逐条 AI 重写
3. 再补触发词和参数

### 17.2 为什么角色总漂

常见原因：

- 角色描述太空
- 世界书太杂乱
- 世界书参数没整理
- 试图让角色承担太多世界观解释任务
- 温度过高

### 17.3 为什么世界书不触发

先查：

1. `keys` 是否太少
2. 触发词是否写错
3. `strategy` 是否不合适
4. 参数是否被 AI 整理得太偏
5. 在试聊区世界书指示有没有变化

### 17.4 为什么小说工坊容易炸

常见原因：

- 文本太长
- 分片太大
- 合并阶段 token 超限
- 模型格式遵守差
- 公益站或代理接口稳定性差

解决思路：

1. 调低分片字数，或改用「按章节」并增大分片章节数
2. 按卷拆跑
3. 先做世界观，再做剧情推进
4. 换更稳的模型

### 17.5 为什么变量很多但不好用

因为变量“多”不等于变量“实用”。

好变量的标准是：

- 能被剧情自然更新
- 有明确用途
- 不会每轮瞎跳
- 不和其他变量重复

## 18. 一个完整实战建议

如果你不知道怎么开始，可以照这个模板做一次：

### 18.1 目标

做一张“赛博朋克情报贩子 + 城市灰区世界观 + 轻度状态追踪”的卡。

### 18.2 操作步骤

1. 在 `角色设定` 里写好角色名、描述
2. 在 `AI 引擎` 里给两个提示词并生成骨架
3. 在 `开场白` 里写好主开场与备选开场
4. 生成 `6` 条世界书骨架
5. 去 `世界书条目` 把人物、地点、组织、规则分别补满
6. 跑 `AI 智能整理`
7. 跑 `AI 触发词生成`
8. 在 `状态栏` 里按人数/预设/生成/排版（视觉主题或自定义）生成变量设计（覆盖 MVU），并注入状态栏
9. 需要时在 `MVU 变量节点` 微调链路或追加单条
10. 在「完成制作 → 角色试聊」里测试五轮
11. 在 `世界书内容监测` 跑一次审计
12. 最后在「角色卡管理」对应卡片底部导出 PNG

这套流程基本就能做出一张“可跑、可修、可继续迭代”的卡。

## 19. 给不同人群的建议

### 19.1 如果你是新手

- 先别碰太复杂的 MVU
- 先学会把角色描述和世界书分开
- 先做一张 5 到 10 条世界书的小卡
- 多试聊，多改开场白和世界书

### 19.2 如果你是设定党

- 少依赖一键成品
- 多用骨架 + 单条重写
- 多用审计和 JSON 实时追踪
- 参数整理放在内容写稳之后

### 19.3 如果你是长篇改卡用户

- 一定要用小说工坊
- 一定要做门控
- 一定要分阶段
- 一定要试“未解锁信息是否泄露”

### 19.4 如果你是机制党

- 上 MVU
- 让变量覆盖真实玩法需求
- 别做纯摆设字段
- 让每个变量都对应明确更新条件

## 20. 最后给一个总原则

这类卡的质量，通常不是由“某一段 description 写得多华丽”决定的，而是由下面四件事共同决定：

1. 角色本体是否清楚
2. 世界书是否分工明确
3. 状态层是否真实可维护
4. 试聊和排错是否认真做过

一句话总结这个项目的正确打开方式：

> 先把角色写清楚，再把世界拆开写，再把状态做成可维护结构，最后用试聊和审计把它打磨成能玩的卡。
